## 6. Task 4.1.1 — Design the Provisioning Record

### Related file

```text
POS.Desktop/Services/Provisioning/ProvisioningRecord.cs
```

### What was done?

A new immutable record was created:

```csharp
public sealed record ProvisioningRecord(
    int? TenantId,
    int? LocationId,
    int? TerminalId)
```

It contains validation helpers:

```csharp
public bool IsValid =>
    TenantId.HasValue && TenantId.Value > 0 &&
    LocationId.HasValue && LocationId.Value > 0 &&
    TerminalId.HasValue && TerminalId.Value > 0;

public bool IsUnprovisioned =>
    TenantId == null && LocationId == null && TerminalId == null;

public bool IsFullyProvisioned => IsValid;

public bool IsHalfProvisioned => !IsUnprovisioned && !IsFullyProvisioned;

public static ProvisioningRecord Unprovisioned { get; } = new(null, null, null);
```

### Yeh kya karta hai?

Yeh terminal ki identity ko safe shape mein hold karta hai:

```text
TenantId
LocationId
TerminalId
```

### Kyun banaya gaya?

POS multi-tenant hai. Local SQLite reads ko pata hona chahiye ke current terminal kis tenant/location/terminal ka data read kar sakta hai.

### Agar na karte to kya hota?

Code direct integer defaults ya fake IDs use kar sakta tha. Is se data leak ka risk hota, especially tenant isolation ke case mein.

### Safe ya risky?

Safe. Record nullable hai aur positive IDs ke bina fully provisioned nahi maanta.

### Real POS benefit

Ek store/branch/terminal sirf apna scoped local data dekh sakta hai. Yeh offline-first POS security ka foundation hai.

## 7. Task 4.1.2 — Implement Real `IProvisionedTerminalContext`

### Related file

```text
POS.Desktop/Services/Provisioning/ProvisionedTerminalContext.cs
```

### What was done?

`IProvisionedTerminalContext` ki real implementation banayi gayi.

Important snippet:

```csharp
private ProvisioningRecord _record = ProvisioningRecord.Unprovisioned;
private readonly object _lock = new();
```

Fail-closed ID getters:

```csharp
public int CurrentTenantId
{
    get
    {
        lock (_lock)
        {
            return _record.IsFullyProvisioned ? _record.TenantId!.Value : 0;
        }
    }
}
```

Same pattern `CurrentLocationId` aur `CurrentTerminalId` ke liye bhi use hua.

State update method:

```csharp
public void UpdateState(ProvisioningRecord record)
{
    ArgumentNullException.ThrowIfNull(record);

    lock (_lock)
    {
        _record = record;
    }
}
```

### Yeh kya karta hai?

Agar terminal fully provisioned hai to real IDs return karta hai. Agar unprovisioned ya half-provisioned hai to `0` return karta hai.

### Kyun banaya gaya?

`NoProvisionedTerminalContext` temporary fail-closed fallback tha. Real POS ko actual tenant/location/terminal context chahiye.

### Agar na karte to kya hota?

Local DB filters hamesha tenant 0 par operate karte. Real scoped data kabhi visible nahi hota.

### Safe ya risky?

Safe because:

```text
Default state = unprovisioned
Invalid/partial state = fail closed
No fake tenant ID
No sensitive data
Thread-safe lock
```

### Real POS benefit

Terminal app safely identify kar sakti hai ke current machine kis tenant/location/terminal ke under kaam kar rahi hai.

## 8. Task 4.1.3 — Load Provisioning State at Startup

### Related files

```text
POS.Desktop/Services/Provisioning/ProvisioningConfigLoader.cs
POS.Desktop/appsettings.json
```

### What was done?

`ProvisioningConfigLoader` add hua jo startup config se provisioning state parse karta hai.

Important snippet:

```csharp
public static ProvisioningRecord Load(IConfiguration configuration)
{
    if (configuration == null)
    {
        return ProvisioningRecord.Unprovisioned;
    }

    var section = configuration.GetSection("Provisioning");
    if (!section.Exists())
    {
        return ProvisioningRecord.Unprovisioned;
    }

    var tenantIdRaw = section["TenantId"];
    var locationIdRaw = section["LocationId"];
    var terminalIdRaw = section["TerminalId"];

    if (string.IsNullOrWhiteSpace(tenantIdRaw) &&
        string.IsNullOrWhiteSpace(locationIdRaw) &&
        string.IsNullOrWhiteSpace(terminalIdRaw))
    {
        return ProvisioningRecord.Unprovisioned;
    }

    int? tenantId = int.TryParse(tenantIdRaw, out var tId) ? tId : null;
    int? locationId = int.TryParse(locationIdRaw, out var lId) ? lId : null;
    int? terminalId = int.TryParse(terminalIdRaw, out var termId) ? termId : null;

    return new ProvisioningRecord(tenantId, locationId, terminalId);
}
```

`appsettings.json` mein placeholder section add hua:

```json
"Provisioning": {
  "TenantId": null,
  "LocationId": null,
  "TerminalId": null
}
```

### Yeh kya karta hai?

Startup par config se tenant/location/terminal IDs read karta hai.

### Kyun banaya gaya?

Milestone 4.2 se pehle durable persistence nahi tha. Isliye config-based temporary startup source banaya gaya.

### Agar na karte to kya hota?

`ProvisionedTerminalContext` exist karta, lekin startup par initialize nahi hota.

### Safe ya risky?

Safe because:

```text
Missing config = unprovisioned
Empty config = unprovisioned
Invalid config = half-provisioned/fail-closed
No real IDs committed
No secrets
```

### Real POS benefit

Later Milestone 4.2 mein persistent provisioning store isi loader pattern ko replace/extend kar sakta hai.

## 4.1 Startup loader service

**File path**

```text
POS.Desktop/Services/Provisioning/TerminalProvisioningStartupLoader.cs
```

**Important snippet**

```csharp
public async Task LoadAsync(CancellationToken cancellationToken = default)
{
    try
    {
        using var scope = _scopeFactory.CreateScope();
        var store = scope.ServiceProvider.GetRequiredService<ITerminalProvisioningStore>();

        var record = await store.GetProvisioningRecordAsync(cancellationToken).ConfigureAwait(false);

        if (record.IsFullyProvisioned)
        {
            UpdateRuntimeContext(record);
            _logger.LogInformation("Terminal provisioning loaded from persistent store. State: provisioned.");
        }
        else
        {
            ResetToFailClosed();
            _logger.LogInformation("Terminal remains unprovisioned.");
        }
    }
    catch (Exception ex) when (ex is not OperationCanceledException)
    {
        ResetToFailClosed();
        _logger.LogError(ex, "An error occurred while loading provisioning state at startup. Terminal remains unprovisioned.");
    }
}
```

### Yeh kya karta hai?

Startup par SQLite se persisted provisioning record read karta hai.

```text
Valid row → runtime context provisioned
No row → fail-closed unprovisioned
Partial/invalid row → fail-closed unprovisioned
Exception → fail-closed unprovisioned
```

### Kyun zaroori?

Restart ke baad singleton context memory mein fresh/unprovisioned hota hai. Loader usko DB se hydrate karta hai.

### Agar na karte to kya hota?

Terminal DB mein provisioned hota, but app restart ke baad runtime filters unprovisioned state use karte.

### Final seeder strategy

Important point: pehle seed implementation ID exist hone par skip kar rahi thi. Usse same DB re-provision case mein new tenant empty catalog dekh sakta tha. Cleanup pass mein final strategy fix hui:

```text
Insert-or-retarget / upsert by Id
```

### Important snippet

```csharp
private async Task UpsertByIdAsync<T>(
    DbSet<T> dbSet,
    IReadOnlyList<T> seedRows,
    CancellationToken cancellationToken)
    where T : LocalCatalogEntity
{
    foreach (var seedRow in seedRows)
    {
        var existing = await dbSet
            .IgnoreQueryFilters()
            .FirstOrDefaultAsync(x => x.Id == seedRow.Id, cancellationToken);

        if (existing is null)
            dbSet.Add(seedRow);
        else
            _dbContext.Entry(existing).CurrentValues.SetValues(seedRow);
    }
}
```

### Yeh kya karta hai?

Seeder:

```text
Row absent ho → insert
Row same Id se already ho → seed values se update/retarget
Same tenant re-run ho → no duplicate
Different tenant re-provision ho → existing rows new TenantId par retarget
```

### Kyun banaya gaya?

Group 1 schema mein bootstrap IDs single-column PK hain. Same Id do tenants ke liye same table mein duplicate nahi ho sakti.

```text
↓

↓
Id already exists → skip
↓
```

## 18. Re-Provisioning Flow

```text
Tenant 42 provisioned
        ↓
Seed rows inserted with TenantId = 42
        ↓
Controlled re-provision to Tenant 99
        ↓
Seeder sees same bootstrap IDs already exist
        ↓

        ↓
Seeder retargets existing rows to TenantId = 99
        ↓
Tenant 99 can see catalog
        ↓
Tenant 42 sees zero rows
```

### Why retarget instead of duplicate?

Because local bootstrap seed uses single-column primary key:

```text
Id = 1, 2, 3...
```

Same IDs cannot exist twice for different tenants in same SQLite table. So this temporary seed represents only the currently provisioned tenant.

### Important snippet — list items

```csharp
public async Task<IReadOnlyList<CatalogItemDto>> ListItemsAsync(
    CatalogItemQuery query,
    CancellationToken cancellationToken = default)
{
    ArgumentNullException.ThrowIfNull(query);
    var limit = Math.Clamp(query.Limit, 1, MaxQueryLimit);

    var q = BuildItemQuery();

    if (query.CategoryId.HasValue)
        q = q.Where(x => x.CategoryId == query.CategoryId.Value);

    if (!string.IsNullOrWhiteSpace(query.SearchText))
    {
        var pattern = $"%{query.SearchText.Trim()}%";
        q = q.Where(x =>
            EF.Functions.Like(x.ItemName, pattern) ||
            EF.Functions.Like(x.ItemCode, pattern));
    }

    return await q
        .OrderBy(x => x.ItemName)
        .Take(limit)
        .ToListAsync(cancellationToken);
}
```

### Important snippet — identifier lookup

```csharp
public async Task<CatalogItemDto?> FindByIdentifierAsync(
    string identifierValue,
    CancellationToken cancellationToken = default)
{
    if (string.IsNullOrWhiteSpace(identifierValue))
        return null;

    return await BuildItemQuery()
        .Where(x => x.IdentifierValue == identifierValue.Trim())
        .FirstOrDefaultAsync(cancellationToken);
}
```

### Important snippet — active/sellable join

```csharp
from item in _db.LocalItems.AsNoTracking()

join variant in _db.LocalItemVariants.AsNoTracking()
        .Where(v => v.IsDefault &&
                    v.IsSellable &&

    on item.Id equals variant.ItemId
```

```js
function escapeHtml(value) {
  return String(value ?? '').replace(/[&<>"']/g, ch => ({
    '&': '&amp;',
    '<': '&lt;',
    '>': '&gt;',
    '"': '&quot;',
    "'": '&#39;'
  }[ch]));
}
```

### Used for

```text
item.itemName
category label
product initial
toast messages
cart item name
cart item unit
```

### Why important?

Catalog data future mein server/sync se aa sakta hai. Agar product name mein `<script>` type text aa jaye aur direct `innerHTML` mein inject ho, XSS risk hota.

### Safe example

Input:

```text
Chocolate <Pack>
```

Without escape:

```html
<div>Chocolate <Pack></div>
```

Browser ise tag samajh sakta hai.

With escape:

```html
<div>Chocolate &lt;Pack&gt;</div>
```

### Real POS benefit

Future real catalog names UI mein safely render honge.

## 20. Important Design Decisions

### 20.1 Read-only catalog service

```text
Decision:
ICatalogService/CatalogService only reads data.

Reason:
4.4 scope is catalog read path, not catalog management.
```

### 20.2 Tenant filter reliance

```text
Decision:
CatalogService does not use IgnoreQueryFilters().

Reason:
Current tenant isolation should happen automatically through PosLocalDbContext query filters.
```

### 20.3 Active/sellable filtering

```text
Decision:
Only active items and active/sellable default variants appear.

Reason:
Blocked/discontinued/non-sellable products should not appear in checkout grid.
```

### 20.4 Bridge DTOs, not EF entities

```text
Decision:
Return DTOs through bridge.

Reason:
Frontend should not depend on EF entity shape.
```

### 20.5 Static arrays removed

```text
Decision:
Remove ITEMS[] and CATEGORIES[].

Reason:
Checkout screen must use SQLite-backed catalog.
```

### 20.6 escapeHtml

```text

Escape catalog strings inserted into innerHTML.

Future synced catalog data may contain characters that should render as text, not HTML.
```

### 20.7 Index-only migration

```text

Add only indexes in AddLocalCatalogSearchIndexes.

Performance improvement without destructive schema changes.
```

```csharp
public sealed class SqliteTestDatabase : IDisposable
{
    private readonly SqliteConnection _connection;
    private readonly DbContextOptions<PosLocalDbContext> _options;

    public SqliteTestDatabase()
    {
        _connection = new SqliteConnection("Filename=:memory:");
        _connection.Open();

        _options = new DbContextOptionsBuilder<PosLocalDbContext>()
            .UseSqlite(_connection)
            .Options;
    }

    public PosLocalDbContext CreateDbContext(
        IProvisionedTerminalContext? terminalContext = null)
    {
        var context = terminalContext ?? new NoProvisionedTerminalContext();
        var db = new PosLocalDbContext(_options, context);
        db.Database.EnsureCreated();
        return db;
    }

    public void Dispose()
    {
        _connection.Dispose();
    }
}
```

### Yeh kya karta hai?

- In-memory SQLite connection open karta hai
- Shared options create karta hai
- Provisioned/unprovisioned DbContext create karta hai
- Schema ensure karta hai
- End par connection dispose karta hai

### Yeh kya karta hai?

### Test flow

```text

        ↓

        ↓
Read as Tenant 42

        ↓

        ↓

        ↓

        ↓

```

### Core helper snippet

```csharp
private async Task VerifyFilterAsync<TEntity>(
    Func<int, TEntity> createEntityFunc,
    Func<PosLocalDbContext, DbSet<TEntity>> getDbSetFunc)
    where TEntity : class
{
    using var dbHarness = new SqliteTestDatabase();

    using (var seedDb = dbHarness.CreateDbContext(
        new NoProvisionedTerminalContext()))
    {
        var entityA = createEntityFunc(42);
        var entityB = createEntityFunc(99);

        seedDb.Set<TEntity>().AddRange(entityA, entityB);
        await seedDb.SaveChangesAsync();
    }

    using (var readDb42 = dbHarness.CreateProvisionedDbContext(42))
    {
        var results42 = await getDbSetFunc(readDb42).ToListAsync();
        Assert.Single(results42);
    }

    using (var readDbUnprovisioned =
        dbHarness.CreateUnprovisionedDbContext())
    {
        var resultsUnprovisioned =
            await getDbSetFunc(readDbUnprovisioned).ToListAsync();

        Assert.Empty(resultsUnprovisioned);
    }
}
```

### Yeh kya prove karta hai?

EF global query filters consistently apply ho rahe hain.

### Important baat

Read path normal `ToListAsync()` use karta hai, `IgnoreQueryFilters()` nahi.

### Real POS benefit

Tenant A ka data Tenant B ko nahi dikhega. Unprovisioned terminal ko kisi tenant ka data nahi dikhega.
